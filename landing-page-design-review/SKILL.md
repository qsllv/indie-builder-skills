# Landing Page Design Review

## Purpose

Use this skill to design or review product landing pages that must explain a product clearly, make value perceptible to the customer, build trust, and drive one primary action.

Treat a landing page as a persuasion path, not a brochure. The page should help visitors answer:

- What is this?
- Is it for me?
- What problem does it solve?
- What result will I get?
- Why should I believe it?
- What should I do next?

## Workflow

1. Identify the page goal, target user, traffic context, and primary CTA. If these are missing, infer reasonable defaults and state assumptions.
2. Clarify the product's core promise using this shape: "Help [target user] achieve [outcome] in [situation] without [pain or friction]."
3. Build or evaluate the page section by section, checking that each section advances one argument.
4. Translate features into perceived value: feature -> behavior change -> business or personal outcome -> emotional relief.
5. Check proof, objections, and CTA strength before judging visual polish.
6. For audits, lead with the highest-impact issues and classify them by priority.

## Design Principles

### Clarity before cleverness

The first viewport must explain the product category, audience, problem, result, and next action. If a visitor cannot understand the page in 5 seconds, the design is failing no matter how polished it looks.

### Value before features

Do not stop at what the product does. Explain what changes for the user after using it. Replace feature lists with outcomes, examples, workflow changes, saved time, avoided risk, higher confidence, or better decisions.

### Specific scenes before abstract claims

Avoid vague claims like "streamline workflows", "empower teams", "AI-powered productivity", "all-in-one solution", and "boost efficiency" unless immediately backed by a concrete scenario or proof.

Prefer copy like:

- "Turn 200 scattered customer messages into a ranked feature backlog."
- "Generate a client-ready weekly report from your Stripe, GA, and database metrics."
- "Find the 3 churn reasons that appear most often in support tickets."

### One job per section

Each section should perform one persuasion job: orient, name the pain, show the solution, prove value, explain the workflow, handle objections, or ask for action. Merge or remove sections that repeat the same job.

### Visuals must carry information

Use product screenshots, result previews, workflow diagrams, before/after states, customer artifacts, or concrete use-case visuals. Decorative illustrations, abstract gradients, and generic stock imagery rarely create perceived value.

### Trust must be earned

Use evidence: real screenshots, numbers, testimonials, case studies, recognizable customers, security details, guarantees, demos, comparison tables, or transparent limitations. Do not rely on self-praise.

### CTA must describe the next valuable step

Avoid weak CTA text such as "Submit", "Learn More", "Get Started", or "Click Here" when a more concrete action is possible.

Prefer:

- "Create My First Report"
- "Analyze My Feedback"
- "See the Product in Action"
- "Start a Free Trial"
- "Upload Data and Preview Results"

## Default Page Structure

Use this structure unless the product or traffic source calls for a different flow:

1. Hero: positioning, outcome, CTA, product or result visual.
2. Problem: the specific pain or cost of the current way.
3. Solution: how the product changes the workflow.
4. Benefits: 3-5 outcomes the user can perceive.
5. How it works: 3-4 concrete steps.
6. Proof: screenshots, metrics, testimonials, examples, logos, case studies.
7. Use cases or audience paths: who uses it and when.
8. Objection handling: price, setup, migration, reliability, security, learning curve, credibility.
9. Final CTA: repeat the core promise and ask for the primary action.

For detailed section patterns, read `references/section-patterns.md`.

## Audit Method

When reviewing an existing landing page, inspect these dimensions before making style suggestions:

1. Positioning clarity
2. Customer relevance
3. Problem specificity
4. Value perceptibility
5. Information hierarchy
6. Proof strength
7. CTA clarity
8. Visual usefulness
9. Differentiation
10. Objection handling
11. Conversion path

Read `references/audit-checklist.md` for the detailed audit rubric and issue rules.

## Output Formats

### When designing a landing page

Return:

- Page strategy: target user, core problem, core promise, primary CTA.
- Section plan: ordered sections with each section's persuasion job.
- Key copy: hero headline, subheadline, CTA, section headings.
- Visual direction: what screenshots, diagrams, demos, or assets are needed.
- Proof plan: what evidence should appear and where.

### When auditing a landing page

Return:

- Overall judgment: the biggest reason the page does or does not work.
- Priority issues:
  - P0: Visitors cannot understand, trust, or act.
  - P1: Clear conversion or persuasion weakness.
  - P2: Polish, hierarchy, or supporting improvements.
- Section-by-section notes.
- Concrete rewrites or structural fixes for the most important sections.

## Quality Bar

A landing page is acceptable only when:

- The hero names the audience, product category or use case, concrete outcome, and next action.
- At least one section makes the customer's current pain feel specific and recognizable.
- Feature claims are translated into customer outcomes.
- The page includes proof proportional to the ask.
- CTA copy tells visitors what they will get or do.
- Visuals help explain the product, result, or workflow.
- The page has one primary conversion path.
